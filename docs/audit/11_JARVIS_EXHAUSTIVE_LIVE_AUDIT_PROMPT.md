# JARVIS — ПОВТОРЯЕМЫЙ ВСЕОБЪЕМЛЮЩИЙ ЖИВОЙ АУДИТ

Скопируй весь текст этого файла в новую задачу Codex/Claude, запущенную на целевой Windows-машине с репозиторием JARVIS. Это не обзор кода и не прогон одних автотестов. Цель — доказать на живой машине, что готовое изделие работает целиком, исправить найденные дефекты по первопричине и оставить воспроизводимые доказательства.

## Директива

Работай автономно до объективного результата `READY` либо до доказанного внешнего блокера. У тебя есть полномочия читать и изменять репозиторий, запускать локальные процессы и Docker, использовать живую настроенную LLM, безопасно обращаться к интернет-источникам, создавать audit-artifacts, исправлять дефекты, коммитить и пушить проверенные изменения прямо в `main`.

Все модельные проверки выполняй только с профилем `qwen36-vl` и живой моделью Qwen. Gemma не запускай, не переключай на неё стек и не включай её ни в operator suite, ни в GUI/API/Telegram/recovery/soak-проверки. Это ограничение заменяет любые требования других audit-документов о втором профиле.

Не подменяй живую проверку автотестами. Автотесты обязательны, но они лишь один слой доказательств. Для каждого пользовательского контура используй настоящий интерфейс, настоящий backend, настоящую настроенную локальную LLM и реальные внешние источники там, где функциональность зависит от интернета.

Не лечи симптомы. Для каждого дефекта найди нарушенный контракт или первопричину, исправь её, добавь регрессионный тест и повтори исходный живой сценарий. Не добавляй ложные ответы, молчаливые fallback, фиктивный успех, бесконечные retry, отключение проверки или особые ветки только ради теста.

## 1. Восстановление и синхронизация

1. Восстанови рабочий контекст из git history, audit artifacts, локальных журналов и сохранённой памяти.
2. Зафиксируй `HEAD`, `origin/main`, dirty/untracked state, активные процессы, контейнеры, модель, порты, профиль, логи и целостность SQLite.
3. Сохрани пользовательские незакоммиченные и untracked-файлы. Не стирай и не перезаписывай чужую работу.
4. Получи актуальный `origin/main`; безопасно синхронизируй рабочую папку. При конфликте не уничтожай локальные данные.
5. До любых рискованных миграций сделай проверяемый внешний checkpoint баз, state, конфигурации и критичных artifacts; запиши SHA-256.
6. Создай новый run в `.audit/runs/<UTC>_<HEAD>/functional/` либо корректно продолжи текущий незавершённый run. Не выдавай artifacts старого HEAD за новые доказательства.

## 2. Полный запуск стека

Подними стек только поддерживаемым launcher-контуром:

- Docker Desktop и dispatcher;
- выбранную локальную LLM;
- backend;
- frontend Command Center;
- host bridge;
- Telegram bridge/bot;
- self-healing/supervisor, если они входят в штатную конфигурацию.

Launcher обязан дождаться настоящей completion-readiness модели до запуска потребителей, которые могут принять пользовательское сообщение. Проверка `/health` или `/v1/models` без живого completion недостаточна.

После запуска докажи:

- каждый нужный порт принадлежит ожидаемому процессу;
- контейнер использует точный ожидаемый image ID/digest и модель;
- `/health`, `/api/status`, OpenAPI и WebSocket доступны согласно контракту;
- живой completion возвращает точный проверочный ответ;
- Telegram прошёл `getMe`, привязан к каноническому bot realm и durable store;
- launcher state соответствует реальным PID/container ID и безопасен после power loss;
- в логах нет segfault, restart loop, OOM, необработанных exception и ложного `ready`.

Любой частичный старт, stale state, неверное владение процессом, fail-open rollback или автоматический restart без восстановления причины — finding и блокер `READY`.

## 3. Слои проверки

Проверяй каждый слой отдельно и в сквозных пользовательских journeys:

1. Unit/contract/regression tests.
2. Полный backend test suite, lint, typecheck и сборка frontend.
3. CLI-команды и негативные варианты аргументов.
4. Каждый OpenAPI operation: авторизация, схема, успешный безопасный путь, отрицательный путь, HITL для мутаций, request ID и отсутствие утечки секретов.
5. Streaming и WebSocket: порядок событий, финализация, reconnect, interrupted turn, отсутствие дубликатов.
6. Реальный GUI через браузер: login/logout, chat, history, stream, trace, memory, files, tools, missions, approvals, diagnostics, admin и ошибки.
7. Telegram: реальный bridge, per-user/per-chat history, owner/guest policy, reboot continuity, `/new`, `/reset`, durable inbox, idempotency и retry.
8. Живая LLM: качество ответа, tool planning, self-check, multi-turn, clarification и деградация.
9. Web/browser и finance: настоящий поиск, свежие первичные источники, точные идентификаторы, значение, валюта/единица и timestamp с timezone.
10. Файлы и документы: ingest, extract, search, compare, generate, download, quarantine и persistence.
11. Missions, approvals, reminders, screen watch, dispatcher control и прочие мутации: ровно один эффект при повторе запроса.
12. Persistence/recovery: рестарт bridge/backend/frontend/model/всего стека, имитация power-loss state, битый/отсутствующий state, незавершённый turn и восстановление SQLite WAL.
13. Performance/stability: latency, concurrency, длинный контекст, bounded soak, память/GPU, отсутствие деградации и restart loop.

## 4. Полная карта функций

Построй `FEATURE_JOURNEY_MAP.csv` из:

- OpenAPI и backend routes;
- CLI parser/subcommands;
- GUI pages, controls и actions;
- Telegram commands и update paths;
- tool registry и authorization classes;
- model profiles и launcher presets;
- storage/migration/recovery paths;
- документации заявленных функций.

Для каждой строки укажи минимум: идентификатор функции, интерфейс, профиль, предусловия, happy path, отрицательный path, restart/persistence relevance, evidence, verdict и finding ID. Ни одна user-facing функция не может остаться `NOT_TESTED` при `READY`. Недоступные по среде функции получают доказанный `BLOCKED_ENV`, но критичный контур с таким статусом блокирует `READY`.

## 5. Operator suite на живой LLM

Профиль `qwen36-vl` проверь минимум 40 различными GUI-кейсами. Все модельные сценарии выполняй только с живой Qwen. Критичные сценарии повтори не менее трёх раз, прочие недетерминированные — не менее двух раз.

Обязательные классы:

- точные короткие ответы и строгий формат;
- полезные развёрнутые ответы;
- русский и английский языки;
- follow-up с местоимениями и ссылками на прошлые сообщения;
- независимые истории двух пользователей и двух чатов;
- clarification при реальной неоднозначности;
- корректный отказ от выдумывания;
- web research и сравнение источников;
- финансы с точным тикером/контрактом/ISIN, freshness, единицей, валютой, timezone и цитатами;
- файлы/документы;
- локальная диагностика и безопасные инструменты;
- approval-gated мутация;
- mission/tool-loop;
- offline/model-down и восстановление без потери turn;
- длинный диалог, перезапуск и продолжение контекста.

Для каждого ответа оцени: соответствие запросу, фактическую точность, полноту, формат, groundedness, tool correctness, безопасность, сохранение контекста, отсутствие внутренних заглушек и отсутствие повторного эффекта.

Неприемлемы: фиктивный успех, ответ из памяти модели вместо обязательного live-source, устаревшая цена без точного времени, неправильный инструмент, потеря контекста, смешение пользователей, скрытая ошибка, бесконечное ожидание, повторная мутация, необоснованный отказ или техническая заглушка вместо ответа.

## 6. Telegram и права

До теста сними безопасный snapshot Telegram identity/store без публикации токенов. Докажи:

- канонический realm имеет вид `telegram:<getMe bot id>`;
- один bot ID не связан с несколькими realm;
- legacy realm мигрируется явно, однократно и без смешения другого бота;
- каждый Telegram user/chat имеет собственную durable conversation;
- reboot bridge/backend не меняет conversation ID;
- role change не стирает историю;
- `/new` и `/reset` — единственные намеренные границы сброса;
- owner и guest не получают чужие данные или права;
- права конкретных учёток соответствуют durable binding, а не username-догадке;
- отсутствующему пользователю права не выдаются;
- transient LLM outage не теряет update;
- permanent poison не блокирует навечно последующие update;
- повтор после потерянного HTTP-ответа не создаёт второй эффект.

Не отправляй реальные пользовательские сообщения без явной необходимости. Если для полноценного E2E нужен настоящий входящий Telegram update, используй согласованный тестовый chat и зафиксируй его отдельно.

## 7. Финансовый контур и интернет

Для котировок, контрактов, ставок и других изменяемых данных интернет-поиск обязателен. Для каждого числового ответа проверь:

- точный инструмент и класс актива;
- provider symbol и официальный identifier;
- значение и валюту;
- единицу измерения из авторитетного источника;
- полный timestamp, timezone и торговую дату;
- freshness относительно текущего времени и режима рынка;
- прямые ссылки на источник значения и источник спецификации;
- отсутствие подмены spot/future, front month/continuous, акции/депозитарной расписки, ticker/ISIN.

Проверь минимум акции, криптоактив, валюту и биржевой товар/фьючерс. При невалидных, неоднозначных, устаревших или недоступных данных система должна fail closed и честно объяснить, чего не хватает.

## 8. Recovery и разрушительные сценарии

Все destructive/recovery-тесты выполняй на контролируемом checkpoint. Обязательно проверь:

- backend/frontend/bridge restart;
- model container restart;
- self-heal после модельного сбоя;
- launcher start/stop/restart/reuse;
- пропавший и повреждённый launcher state;
- stale PID/container ID;
- concurrent replacement/cutover с compare-and-swap;
- candidate image/model, который не проходит живой completion;
- rollback к точному прежнему image ID и модели;
- отсутствие удаления чужого replacement-контейнера;
- SQLite integrity/FK/WAL после жёсткого завершения процесса;
- сохранение Telegram history и незавершённого update.

Никогда не удаляй живые данные ради теста. Восстановление считается успешным только после повторной живой проверки LLM, API, GUI и Telegram store.

## 9. Findings и исправления

На каждый воспроизводимый дефект создай `.audit/.../findings/FUNC-NNNN.md`:

- severity и затронутый контур;
- точные шаги воспроизведения;
- expected/actual;
- первопричина;
- исправление контракта;
- автоматический regression;
- живой retest;
- evidence paths;
- commit SHA;
- окончательный verdict.

После каждого исправления запускай сначала минимальный regression, затем затронутый набор, затем полный набор в объёме риска, затем исходный живой сценарий. Не закрывай finding только по зелёному unit-тесту.

## 10. Evidence и контроль прогресса

Храни machine-readable evidence без секретов:

- JSONL запросов/ответов и timing;
- CSV/JSON feature matrix;
- manifests с HEAD, profile, image ID, model, timestamps и SHA-256;
- GUI screenshots и DOM/visible-state evidence;
- sanitized logs;
- тестовые отчёты;
- DB integrity/binding snapshots;
- recovery timelines;
- findings и retest records.

Веди `JOURNAL.md`, `FUNCTIONAL_STATE.json` и `PIPELINE_STATE.json` после значимых этапов. Отчитайся пользователю примерно на 50% и 90%, а также немедленно при P0, риске данным или внешнем блокере. Не засоряй чат обычным ходом работы.

## 11. Commit и push

Перед коммитом:

1. Убедись, что не затронуты пользовательские untracked/несвязанные файлы.
2. Проверь `git diff`, `git diff --check`, lint, полный backend suite, frontend tests/typecheck/build.
3. Повтори живой LLM smoke и критичные E2E.
4. Коммить только проверенные изменения с понятным сообщением.
5. Синхронизируйся с `origin/main` безопасным способом и пушь в `main`.
6. После push зафиксируй точный remote SHA и повтори короткий post-push smoke на нём.

Audit artifacts, содержащие секреты или реальные персональные данные, не коммить. Секреты всегда редактируй в evidence.

## 12. READY gate

Разрешено объявить `READY` только одновременно при следующих условиях:

- проверена вся карта user-facing функций, нет `NOT_TESTED`;
- выполнен минимум operator suite на `qwen36-vl`; другие модельные профили не тестируются;
- пройдены GUI, API, CLI, Telegram, web/finance, files, missions, approvals, persistence, recovery и bounded soak;
- полный автоматический набор зелёный;
- живые LLM-проверки зелёные;
- все P0/P1 и все воспроизводимые product-blocking дефекты исправлены и живо перепроверены;
- нет открытой угрозы данным, смешения пользователей, повторных мутаций, ложной готовности или fail-open recovery;
- artifacts относятся к фактически протестированному итоговому commit;
- итоговый commit находится в `origin/main`;
- полный стек оставлен в ожидаемом рабочем состоянии.

Если хотя бы один пункт не выполнен, итог — `NOT READY` с точным остатком, evidence и следующим безопасным действием. Не округляй прогресс до 100% заранее.

## 13. Итоговый отчёт

Сообщи кратко:

- `READY` или `NOT READY`;
- протестированный commit и remote SHA;
- профиль `qwen36-vl`, модель Qwen и image ID;
- число функций, operator cases/repeats, API operations и findings;
- автоматические и живые результаты;
- выполненные recovery/soak сценарии;
- исправленные P0/P1 с commit SHA;
- путь к audit run и главным evidence;
- состояние полного стека;
- оставшиеся ограничения, если они есть.

Не заявляй, что «всё работает», без этой доказательной базы.
